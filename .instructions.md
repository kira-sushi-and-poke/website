# Coding Instructions for Kira Sushi & Poke Website

## Code Quality Standards

### DRY Principle (Don't Repeat Yourself)
- Extract any code duplicated 2+ times into shared utility functions
- Create constants for any value used 3+ times across the codebase
- Use shared utilities from `src/lib/` instead of inline implementations

### Documentation
- Add JSDoc comments for all exported functions
- Include parameter types, return types, and descriptions
- Document complex logic with inline comments

### Code Organization
- Place shared utilities in `src/lib/`
- Use semantic file and folder names
- Keep files focused on a single responsibility

## Code Style

### Quotes
- **Always use double quotes (`"`)** for strings when possible
- Use double quotes for imports, JSX attributes, object keys, etc.
- Only use single quotes when necessary (e.g., strings containing double quotes)

### Naming Conventions
- Use `camelCase` for variables and functions
- Use `PascalCase` for React components
- Use `UPPER_SNAKE_CASE` for constants
- Prefix boolean variables with `is`, `has`, `should`, `can`
- Use descriptive names that explain purpose

### Function Structure
- Keep functions small and focused (ideally < 50 lines)
- Return early to reduce nesting
- Use destructuring for cleaner code
- Validate inputs at the start of functions

## Next.js and React Patterns

### Server vs Client
- Use `"use client"` directive for client components (interactive, useState, useEffect, etc.)
- Use `"use server"` directive for server actions and API utilities
- **Always add `"use server"` to files that access sensitive env variables** (like `process.env.ACCESS_TOKEN`)
- **Prefer Server Actions over API routes** whenever possible (simpler, type-safe, fewer files)
- Only use API routes for external webhooks or when you need custom HTTP handling

### Server Actions vs API Routes
- ✅ **Use Server Actions for:**
  - Internal operations called from your app
  - Operations that can throw errors and be caught by the client
  - When you want direct function calls instead of fetch
  - Type-safe communication between client and server
- ❌ **Use API Routes for:**
  - External webhooks (e.g., Square payment webhooks)
  - Third-party integrations that POST to your endpoints
  - When you need custom headers or specific HTTP status codes

### Component Best Practices
- Extract reusable logic into custom hooks
- Use React.memo for expensive components
- Prefer composition over prop drilling
- Keep components focused on a single concern

## Square API Patterns

### Use Shared Utilities
- **Always use `fetchSquare()` helper** from `@/lib/squareApi` for Square API calls
- **Always use `SQUARE_API_VERSION` constant** from `@/lib/constants`
- Never hardcode API version strings or base URLs

### Customer and Order Handling
- Use `splitName()` from `@/lib/splitName` for name parsing
- Use `validateContactDetails()` from `@/lib/validation` for form validation
- Use `generatePickupTimes()` from `@/lib/generatePickupTimes` for time slots

### Error Handling
- Always wrap API calls in try/catch blocks
- Provide user-friendly error messages (not raw API errors)
- Log errors to console for debugging but don't expose sensitive data
- Handle specific error codes (card declined, insufficient funds, etc.)

### Idempotency
- Always include `idempotency_key: crypto.randomUUID()` for Square create/update operations
- Prevents duplicate charges and orders

## Validation Patterns

### Form Validation
- Use shared validators from `@/lib/validation`
- Validate on both client and server side
- Clear field errors on input change
- Show specific error messages per field

### Required Fields
- Always validate required fields before API calls
- Use consistent error messages across the app
- Return early if validation fails

## Environment Variables

### Configuration
- Use `process.env.API_BASE_URL` for Square API base URL (configurable for sandbox/production)
- Never hardcode URLs, tokens, or API keys
- Document all required env vars in `.env.local` or README

### Security
- Never expose sensitive env vars to client (no `NEXT_PUBLIC_` prefix for tokens)
- Always use server-side routes for authenticated API calls
- Add `"use server"` to modules accessing sensitive env vars

## File Structure Conventions

```
src/
  lib/                    # Shared utilities
    constants.js          # App-wide constants
    squareApi.js          # Square API helper
    validation.js         # Form validation utilities
    splitName.js          # Name parsing helper
    generatePickupTimes.js # Time slot generator
  app/
    api/                  # API routes (server-side)
    [feature]/            # Feature-based organization
  components/             # Shared React components
```

## Common Pitfalls to Avoid

- ❌ Don't duplicate validation logic across components
- ❌ Don't hardcode the Square API version
- ❌ Don't expose sensitive data to client-side code
- ❌ Don't forget to handle edge cases (single-word names, empty strings, etc.)
- ❌ Don't use single quotes when double quotes work
- ❌ Don't repeat code - extract to utilities
- ❌ Don't make multiple identical API calls - extract to helper functions

## Testing Considerations

- Test validation logic with edge cases
- Test error handling paths
- Verify sensitive data never reaches client bundle
- Test both Square sandbox and production modes
